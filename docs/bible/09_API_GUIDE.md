# 09 — API Guide

Reference for every HTTP endpoint in the APForce backend. Generated by reading the Express route
definitions directly (`src/app.js` mount points + every file in `src/routes/`) — not inferred from
frontend usage. If behavior described here disagrees with the code, the code wins; re-derive this
doc from source rather than trusting stale prose.

---

## Conventions

**Base URL / mount pattern.** Every route file is mounted under `/api/<resource>` in `src/app.js`.
Middleware applied at the `app.use()` call wraps the *entire* router — individual routes inside the
file may add more (e.g. `checkRole([...])`) but cannot remove what's applied at mount time.

**Auth mechanism.** JWT, read by `authMiddleware` (`src/middleware/auth.js`) from, in order:
1. `req.cookies.accessToken` (httpOnly cookie, set by `/api/auth/login` etc.)
2. `Authorization: Bearer <token>` header (fallback)

Decoded payload becomes `req.user`: `{ id, email, role, name, companyId, plan, planStatus, trialEndsAt }`.
A `temp: true` token (issued mid-2FA-flow) is rejected by `authMiddleware` outside `/verify-totp*`.
`req.user.companyId` is the tenant scope — every multi-tenant query keys off it. `role === 'superadmin'`
bypasses `checkRole()` and `subscriptionMiddleware` entirely (platform staff / support access).

**Role gate.** `checkRole([...roles])` — 403 `Insufficient permissions` if `req.user.role` isn't in the
list (superadmin always passes). `adminMiddleware` is a fixed shorthand for `checkRole(['admin', 'superadmin'])`.
`platformAdminMiddleware` requires `role === 'superadmin'` exactly (used only by `/api/platform`).

**Subscription gate.** `subscriptionMiddleware` — 402 with `code: 'ACCOUNT_SUSPENDED'` or
`code: 'TRIAL_EXPIRED'` if the company's plan is blocked. Bypassed for superadmin and `plan === 'internal'`.
Applied at mount time to routes that accept writes: `metrics`, `admin`, `crm`, `automations`, `campaigns`.
Read-only routers (`analytics`, `audit`) are intentionally left open so a lapsed trial can still view data.

**Standard error envelope.** No single shared error-response builder — routes call
`res.status(N).json({ error: '...' })` directly, and the global handler (`src/middleware/errorHandler.js`)
catches anything thrown/passed to `next(err)`:

| Source | Shape |
|---|---|
| Zod validation failure | `{ error: 'Validation failed', details: err.errors }` (400) |
| Expired/invalid JWT | `{ error: 'Token expired' }` / `{ error: 'Invalid token' }` (401) |
| Uncaught error | `{ error: err.message ?? 'Internal server error', timestamp }` (`err.status ?? 500`) |
| Most route-level errors | `{ error: '<condition-specific message>' }` — status set per-call, no fixed schema beyond `error` |

Success responses are inconsistent by convention, not by design flaw: most return `{ success: true, ... }`,
some (older/legacy endpoints) return the payload directly (see `GET /api/auth/me`, `GET /api/points/leaderboard`).
Check the specific endpoint below rather than assuming a shape.

**Pagination.** No global convention. Three patterns coexist: `page`/`pageSize` query params with a
`{ total, page, pages }` envelope (`crm.js`, `contacts.js`); cursor-based `before`/`nextCursor` for message
threads (`crm.js` lead detail); unpaginated full-scan capped at a fixed `Limit` (kanban view, `500`).

**Rate limiting.** `rateLimit(maxRequests, windowMs)` (`src/middleware/rateLimiter.js`) is applied
per-route, not globally — presence/absence and limits vary by endpoint (see tables below). Login has a
separate per-email `loginRateLimiter` (429 after repeated failures, resets on success).

### ADR-012 — outbound WhatsApp sends (ENFORCED)

Every new send path MUST call a method on `WhatsAppSendService` (`WASendSvc.sendText/sendTemplate/
sendInteractive/sendMedia(...)`) — never `axios`/`fetch` directly to `graph.facebook.com/*/messages`.
Route handlers validate input, call the service, return the result. See `docs/adr/ADR-012-whatsapp-send-service.md`
and `CLAUDE.md` for the full rule. **Known exception found in this audit:** `whatsapp.js:2478`
(`POST /api/whatsapp/inbox/:leadId/mark-read`) posts a read-receipt status update directly to
`graph.facebook.com/.../messages` — a literal violation of the "no direct call" rule, though narrower in
scope than a message send (it's a status ack, not new message content, and writes no DDB message record).
Flag this if extending that endpoint; do not copy the pattern into new code.

### ADR-013 — customer identity (ENFORCED, NOT YET INTEGRATED)

Every new customer-creating entry point MUST call `CIS.resolveOrCreate()` (`src/services/CustomerIdentityService.js`)
— never write `LEAD# METADATA` directly or query `company-phone-index` ad hoc for dedup. **As of this audit,
zero route files import `CustomerIdentityService` or call `CIS.resolveOrCreate()`** — this matches ADR-013's
own text ("exists, not yet integrated"). All current lead-creation paths replicate CIS's dedup logic inline
(GSI query on `company-phone-index` + `to10Digit()` normalization) correctly in most places, but the
atomic `LEAD_PHONE#` lock CIS provides does not exist anywhere in the route layer yet. See per-resource notes
below for which paths are GSI-correct-but-not-CIS vs. genuinely non-compliant (in-memory scan dedup).

## Auth — `/api/auth`

Login, 2FA (TOTP + backup codes), token refresh, employee registration, self-service company signup. No
`authMiddleware` at mount (`app.use('/api/auth', authRoutes)`) — each route opts in individually.

| Method & Path | Auth | Request | Response | Notable errors |
|---|---|---|---|---|
| `POST /login` | Public | `{ email, password }` | No 2FA: `{ success, token, user }` + sets `accessToken`/`refreshToken` cookies. Has 2FA: `{ requiresTOTP: true, tempToken, message }` | 401 invalid creds / inactive account; 429 rate-limited (per-email) |
| `POST /verify-totp` | `tempToken` (from login) | `{ tempToken, totpCode }` | `{ success, token, user }` + cookies | 401 invalid/expired temp token or wrong code (`attemptsRemaining` in body) |
| `POST /verify-totp-backup` | `tempToken` | `{ tempToken, email, backupCode }` | `{ success, token, user, backupCodesRemaining, warning? }` | 401 invalid code/session; 400 no backup codes configured |
| `POST /refresh` | `refreshToken` cookie | — | `{ success, token, user }` + new cookies | 401 missing/invalid/expired refresh token or inactive user |
| `POST /register` | `authMiddleware`, admin/superadmin only (checked in-body, not `checkRole`) | `{ email, password, name, role, mobileNumber?, panNumber?, aadhaarNumber?, homeAddress? }` (Zod `registerSchema`) | `201 { success, message, user }` | 403 non-admin caller; 400 email already registered in this company |
| `POST /logout` | `authMiddleware` | — | `{ success, message }`, clears cookies | — |
| `POST /company-signup` | Public | Zod `companySignupSchema`: `{ companyName, broker, city, adminEmail, adminName, password, adminMobile? }` | `201 { success, message, token, user, company }` — creates `COMPANY_PROFILE` + admin user, 14-day trial | 409 email already registered |
| `GET /me` | `authMiddleware` | — | Full employee record minus `password`/`totpSecret`/`backupCodes`, plus `token` if cookie present (no `success` wrapper) | 404 user not found |

Notes: `planStatus`/`trialEndsAt`/`plan` are embedded in the JWT at issue time (`attachPlan()`) so
`subscriptionMiddleware` never needs a DB round-trip. No `/setup-2fa` route exists here — 2FA enrollment
is admin-initiated via `POST /api/admin/employees/:id/setup-2fa` (see Admin section), not self-service.

**ADR-013 note:** `POST /leads` and `POST /import` both compute `phoneNorm = to10Digit(phone)` and dedup
via the `company-phone-index` GSI (or, for `/import`, an in-memory map built from a full `scanAllLeads()`
pre-pass) — correct normalization, but neither calls `CIS.resolveOrCreate()`, so the atomic `LEAD_PHONE#`
lock is absent. `/import`'s in-memory-scan dedup is the exact gap CLAUDE.md cites at `crm.js:841` — a
concurrent import batch can still race past it.

### Webhook (public — Meta calls these directly)

| Method & Path | Request | Response | Notes |
|---|---|---|---|
| `GET /webhook` | Query: `hub.mode, hub.verify_token, hub.challenge` | Echoes `challenge` (200) or 403 | Meta subscription verification |
| `POST /webhook` | Meta webhook payload | Always `200` (Meta retries on non-200) | Handles `messages` field (inbound text/media + delivery/read status updates) and `message_template_status_update` field. See ADR-013 note below. |

**ADR-013 gap confirmed at this exact line:** in the `POST /webhook` inbound-message handler, the
"known lead" branch (`if (lead) {...}`) resolves via the `company-phone-index` GSI correctly, but the
"unknown contact" branch (`else {...}`, `whatsapp.js` ~line 1408–1470) writes directly to
`INBOX#${companyId}#${phone10}` via `dynamodb.update()` with no phone lock and no `CIS.resolveOrCreate()`
call — this is the item CLAUDE.md's migration table cites as `whatsapp.js:1360`. Two concurrent inbound
messages from a brand-new number can race here. The welcome-message send inside that same branch (on
first contact) correctly uses `WASendSvc.sendTemplate()`.

**ADR-013 gap (not in CLAUDE.md's migration list — new finding from this audit):** both `POST /:id/submit`
and `POST /meta-leads/webhook` write `LEAD# METADATA` items **directly** via `dynamodb.put()` in the route
handler (`forms.js:156` and `forms.js:301`). Both correctly dedup via the `company-phone-index` GSI on
`phoneNorm` first, so the normalization rule (Rule 1) is honored — but neither calls `CIS.resolveOrCreate()`,
so there is no atomic `LEAD_PHONE#` lock. A form flooded with rapid duplicate submissions, or a Meta Lead
Ads webhook retry landing concurrently with a real submission, can race past the GSI check and create two
lead records for the same phone. Given `POST /:id/submit` is a public, unauthenticated, no-rate-limit
endpoint, this is a more exposed instance of the same class of gap already tracked for `whatsapp.js:1360`.
