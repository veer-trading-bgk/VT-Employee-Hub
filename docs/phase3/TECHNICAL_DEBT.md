## Embedding Provider Timeout ‚Äî Alerting Gap for a Pure Connection-Level Outage

**Context (2026-07-15):** `EmbeddingService.embed()` calls Voyage on the live WhatsApp turn's critical path (its vector feeds the prompt via `ConversationalAgentService._fetchKnowledgeContext`). A morning of intermittent Voyage latency produced ~27% query-embed timeouts for `viir_trading` (6 of 22 attempts; 0 the prior day), each firing a `logger.error` ‚Üí Telegram "Production Error" page even though the turn recovered fine (callers degrade to keyword matching). Two things were fixed: the timeout was cut 10s‚Üí5s (moved to `embeddingConfig.js.timeoutMs`, ADR-017 home) to halve the worst-case dead wait, and the catch path was changed to route by **error class** ‚Äî a response-bearing error (401 bad key, 429, 4xx/5xx) still pages immediately via `logger.error`; a timeout / connection failure (no HTTP response) logs `logger.warn` only (CloudWatch, no page).

**Why not a volume tripwire:** the first attempt used an in-memory "‚â•5 failures in 10 min" counter to page on a sustained outage. An adversarial multi-agent review (9 agents, 5 confirmed findings) showed it can essentially never fire at this service's real traffic (~1‚Äì2 embeds/hour, per-warm-Lambda-container counter that a 10-min window and cold-start reclaim both defeat) ‚Äî so it would have silently swallowed exactly the hard failures the old `logger.error` used to page on. Removed in favor of the error-class split above.

**Issue (residual gap):** a *pure connection-level* Voyage outage ‚Äî where requests time out with no HTTP response at all, rather than returning a 5xx ‚Äî now only produces `warn` logs and does **not** page. Most real provider faults (bad/rotated key, rate-limit, server errors) return a response and still page on the first occurrence; a total connection blackout is the one class that goes to CloudWatch only.

**Fix:** add a CloudWatch metric-filter + alarm on the `EmbeddingService: embed timed out / unreachable` warn line (e.g. alarm on N such lines in M minutes). This is the correct cross-container, cold-start-durable signal for a timeout-based outage ‚Äî an ops/infra config, not in-process state, which is precisely why the in-memory counter was the wrong tool.

**Priority:** Low ‚Äî retrieval degrades gracefully to keyword matching during such an outage (customers still get replies), and the more common response-bearing faults already page. This only closes the blind spot for a total Voyage connection blackout.

## GSI Pollution ‚Äî leadsByCompany

**Issue:** `leadsByCompany` GSI contains METRICS, CONV, CONTACT, and CONFIG records because all share the `companyId` attribute. The GSI indexes every record that has `companyId`, not just leads.

**Fix:** Add an `entityType` attribute to all DynamoDB records (e.g. `LEAD`, `CONV`, `CONTACT`, `CONFIG`, `METRIC`). Recreate or add a filtered GSI that keys on `companyId` with a filter condition on `entityType = 'LEAD'`.

**Priority:** Medium ‚Äî must be resolved before the table reaches ~10,000+ records to avoid query performance degradation and excessive read costs from filtering noise out of GSI results.

## `metrics.js`'s `/my-team` route is an unindexed, cross-company full-table scan

**Issue:** `GET /api/metrics/my-team` (`checkRole(['team_lead'])`) resolves a team_lead's direct reports via `dynamodb.scan({ TableName: EMPLOYEES, FilterExpression: 'teamLeadId = :tlId AND #s <> :inactive', ExpressionAttributeValues: { ':tlId': req.user.id, ... } })` ‚Äî a `scan`, not a `query`, with **no `companyId` key condition or filter clause at all**. It reads every employee row in the entire EMPLOYEES table, across every company, and relies solely on `teamLeadId` being a globally-unique employee id (never company-scoped by construction, since ids are opaque UUIDs) to keep one company's team_lead from ever seeing another company's data in the result ‚Äî correct today only because no two companies can share an employee id, not because the query itself is tenant-scoped. Found 2026-07-13 while implementing `TeamScopeService.getTeamMemberIds()` for Contacts team-scoping (OQ-006) ‚Äî that new service deliberately does **not** copy this shape; it queries the EMPLOYEES table's `companyIdIndex` GSI (indexed on `companyId`) first, then filters `teamLeadId` in memory.

**Fix (suggested, not applied in this batch):** migrate `/my-team` to call `TeamScopeService.getTeamMemberIds(req.user.companyId, req.user.id)` instead of its own inline scan ‚Äî same underlying data (active employees whose `teamLeadId` matches the caller), correct indexed query shape, and removes a second, divergent implementation of the same lookup. Not migrated here ‚Äî this finding is a log-only flag per instruction, scoped strictly to the Contacts team-scoping work; `/my-team` itself is untouched.

**Priority:** Low/Medium ‚Äî no confirmed data leak (relies on UUID uniqueness holding, which it does, not on a broken tenant boundary), but a full unindexed table scan on every call is a real, avoidable cost that grows with total platform-wide employee count, not just the caller's own company size, and the pattern is exactly the kind of divergent-implementation risk `TeamScopeService` exists to prevent going forward.

## Public API ‚Äî Form-Submission Endpoint (shipped 2026-07-14)

**What shipped:** A public, API-key-authenticated endpoint ‚Äî `POST /api/public/form-submission` ‚Äî that lets a company's own landing-page backend notify APForce when a lead submits a form, so APForce records the form data on the lead and fires a WhatsApp confirmation via the existing Automation engine. Full spec: `APForce_Public_API_Spec` (Draft v1). Client-facing contract: `docs/PUBLIC_API.md`. Scoped narrowly (spec ¬ß1): one endpoint, one new trigger type (`form_submitted`), reusing every send/identity/automation mechanism that already existed (CIS/ADR-013, WhatsAppSendService/ADR-012, AutomationEngine).

**New surfaces:**
- `src/services/ApiKeyService.js` ‚Äî generate / verify / list / revoke. SHA-256 hash at rest, raw key returned once and never stored or logged.
- `src/middleware/apiKeyAuth.js` ‚Äî `X-API-Key` ‚Üí verify ‚Üí `req.company` (NOT `req.user`; no session).
- `src/middleware/rateLimiter.js` ‚Äî added `apiKeyRateLimit()` (60/min per key), reuses the existing `atomicIncrement` primitive.
- `src/routes/public.js` ‚Äî the endpoint. `src/routes/apiKeys.js` ‚Äî admin generate/list/revoke (session-auth, admin-only).
- Settings ‚Üí API Keys section (admin-only) + automation canvas `form_submitted` trigger option.

**Four design decisions taken during build (all confirmed with Viir before writing):**
1. **Key‚Üícompany resolution via an O(1) lookup item.** The main record is `CONFIG#APIKEY#{companyId}/KEY#{keyId}`, but `verify()` knows only the raw key, not the companyId. A companion item `CONFIG#APIKEY#LOOKUP#{keyHash}/LOOKUP ‚Üí {companyId, keyId, status}` is written atomically alongside it (TransactWrite) so verification is a single GetItem by hash ‚Äî avoiding a full-table Scan of the shared METRICS table on every public call. Both records revoke together.
2. **Claim-first idempotency (not the spec's literal "write after success").** The `IDEMP#{companyId}/{idempotencyKey}` record is claimed atomically (`attribute_not_exists`) *before* processing ‚Üí 409 on conflict. This is what actually prevents a double-click / slow-network retry from firing the confirmation template twice; a check-then-write-after-success leaves a TOCTOU window where both near-simultaneous submits pass the check. On a *processing* failure the claim is released so a genuinely failed submit stays retryable; once CIS commits, the claim is never released. 24h TTL.
3. **Trait tokens are namespaced `{{trait.<key>}}`.** Resolved in `welcomeVariables.js` (both the template-params path and the free-text path), so they never collide with the fixed `{{name}}/{{phone}}/{{source}}` registry, and `findUnsupportedTokens()` still rejects genuinely-unknown tokens while allowing any trait the client sent.
4. **Traits are stored on the interaction/touch metadata**, via CIS's existing `metadata` passthrough (`formTraits`), keeping CIS unmodified (spec ¬ß4) ‚Äî they are visible in the lead's activity timeline, and separately carried in the automation context for `{{trait.*}}` resolution. They are **not** a first-class, queryable lead attribute (see debt below).

**Known residual debt / limitations:**
- **Traits are not queryable as a lead field.** Decision #4 stores them on the interaction metadata, not the LEAD# item. If a future need arises to filter/segment leads by a form trait, a persistent `formTraits` map on the lead record (a post-CIS write) would be required ‚Äî deliberately deferred.
- **`idempotencyKey` is required.** The endpoint depends on the client sending a stable key per logical submission for its double-submit guarantee to hold; a client that sends a fresh random key per retry gets no protection. Documented in `docs/PUBLIC_API.md`.
- **Accepted limitation ‚Äî a hard process/Lambda kill between the claim write and the complete-mark strands the claim.** The claim is released on a *caught* failure (`CIS.resolveOrCreate` throwing ‚Üí `catch` deletes the claim), but a hard kill (Lambda timeout/OOM/crash) after the claim `put` succeeds and before either CIS commits or the complete-mark runs is not caught by any `catch`, so the `IDEMP#` record is left in `status: 'processing'` and blocks a retry of that exact `idempotencyKey` until its **24h TTL** expires. This is deliberately **accepted, not fixed**: the safe alternative (release-before-commit) would re-open the double-fire race the claim-first design exists to close, and the realistic blast radius is one stranded submission recoverable by the client with a new `idempotencyKey` or a 24h wait. A future hardening (a short claim TTL extended to 24h only on completion, so a stranded `processing` claim self-clears in minutes) is noted here should the tradeoff ever need revisiting.
- **Out of scope (spec ¬ß11), tracked in `PENDING_WORK.md`:** multiple keys per company, key rotation policy, and per-key custom rate limits ‚Äî deferred until a real client's usage pattern is known.

**Validation:** `tests/apiKeyService.test.js`, `tests/apiKeyAuth.test.js`, `tests/publicFormSubmission.test.js`, `tests/formTraitVariables.test.js` ‚Äî all green; full suite 1620/1620 passing, frontend `tsc --noEmit` clean. Not yet exercised against a real browser/live key (no staging key issued at build time).

**Priority:** Shipped feature ‚Äî this entry documents the design rationale and the deferred items, not an open defect.

## Inbox mobile Back button ‚Äî two real-device bugs (fixed 2026-07-14)

Both reported via real-device testing (not harness mock data), on `dashboard/src/app/(v3)/inbox/page.tsx`'s mobile conversation view.

**Bug 1 ‚Äî Back button intermittently disappears/reappears.** *Root cause:* the Back button was nested INSIDE the thread-column `<div>` whose mobile visibility is `snapshotOpen ? 'hidden xl:flex' : 'flex'`. Opening the Info/details panel (ThreadPane's avatar/Info tap ‚Üí `onOpenSnapshot` ‚Üí `setSnapshotOpen((o) => !o)`) set `snapshotOpen = true`, which flips the whole thread column to `hidden` below `xl` ‚Äî taking the Back button with it. Closing Info, or a page refresh (which resets `snapshotOpen` to its `useState(false)` default), brought it back. So the "intermittent disappearance" was `snapshotOpen` flapping, not a permanent gate.

**Bug 2 ‚Äî "present for admin, absent for employee."** *Root cause: NOT a role gate.* The Back button's render condition (`activeConv`, `snapshotOpen`, the `md` breakpoint) has zero role inputs ‚Äî verified: `CommunicationsContent` (the parent that renders Back) contains no role/auth logic, `snapshotOpen` is a literal `useState(false)` with no role-conditional/lazy initializer and no auto-open effect, and the `(v3)` layout adds no role-based mobile branch. The admin/employee correlation was a **data/auto-select artifact**: the backend inbox shows admins/managers all leads + unknowns but employees only their own assigned leads (`whatsapp.js` `/inbox`, `canViewAll` filter), so the auto-open effect reliably lands admins in a thread (Back present) while an employee with an empty/unopened filtered inbox has `activeConv === null` ‚Üí the thread column's empty-state branch (`hidden ‚Ä¶ md:flex`) renders, which has no Back on mobile because there is genuinely no open thread. (Opening Info ‚Äî bug 1 ‚Äî was the other way it could appear "missing.")

**Fix (one change resolves both):** the Back button was hoisted OUT of the `snapshotOpen`-gated thread column into a mobile-only bar rendered ABOVE the columns, gated ONLY on `activeConv` (+ `md:hidden`). Implemented as an outer `flex-col` wrapper containing the Back bar + the original (unchanged) inner `flex-row` of the three columns ‚Äî so the 3-pane desktop layout, the snapshot panel's `w-[320px]`, and its `xl:flex`/`xl:block` toggle behavior are all byte-for-byte unchanged; only Back's own gating moved. Result: Back is now pure navigation (thread/snapshot ‚Üí list), present on mobile whenever a conversation is open, identical for every role and regardless of whether the Info panel is open.

**Validation:** `tsc --noEmit` clean; structural trace across all three breakpoints (`<md`: Back present in both thread and Info views, `activeConv`-gated only; `md‚Äìxl` and `xl+`: Back hidden, 3-pane and snapshot toggle unaffected). Real-device breakpoint rendering not performed in-session (no working E2E login this environment) ‚Äî manual on-device confirmation at `<md`/`md`/`xl` still recommended, per the same limitation noted elsewhere in this doc.

**Priority:** Fixed ‚Äî was a genuine, twice-reported daily-use UX bug (Back button is the only way back to the conversation list on mobile).

## INCIDENT #2 ‚Äî Nova broke every turn AFTER the first: Bedrock Converse requires a user-first message array (2026-07-14)

**Severity:** Production, customer-facing. The AI answered a customer's first message but went **silent on every subsequent message** in the same conversation. Distinct root cause from Incident #1 (that was IAM; this is message-shape) ‚Äî surfaced only once the IAM fix let real turns through.

**Symptom.** Turn 1 gets an AI reply; turn 2 onward gets nothing ‚Äî no reply, no error to the user, just silence. Reproduced live on conv `conv_01KX19K3W0CZ4KSSB1MTQ8JRYP` (lead `28899a88`, 9886141993): turn 1 at 11:06 succeeded (two Nova `AIUSAGE#` writes); turns 2/3/4 at 11:07:58 / 11:08:22 / 11:08:40 each logged `ConversationalAgentService: generate failed ‚Ä¶ provider_error ‚Äî A conversation must start with a user message. Try again with a conversation that starts with a user message.` and wrote no `AIUSAGE#`.

**Root cause ‚Äî an Anthropic-vs-Bedrock message-shape assumption difference.** `AIService` builds the provider message array as `[...conversationHistory, { role: 'user', content: promptText }]` (`AIService.js:129`), and `ConversationalAgentService._fetchConversationHistory` maps each stored `MSG#` by direction (`inbound‚Üíuser`, `outbound‚Üíassistant`, `ConversationalAgentService.js:185-188`). The **first message ever written under a freshly-created lead is the AI's own turn-1 reply** (an *outbound* ‚Üí `assistant`) ‚Äî the customer's turn-1 inbound is not stored under that `leadPK` because the lead did not exist yet when it arrived. So from turn 2 on, `messages[0].role === 'assistant'`. The **Anthropic Messages API tolerated an assistant-first history; Bedrock Converse hard-rejects it** ("A conversation must start with a user message"). The rejection throws before `_logUsage()`, so ‚Äî exactly like Incident #1 ‚Äî the missing `AIUSAGE#` was the *symptom*, not a logging bug. Turn 1 worked only because a brand-new lead has empty history, so its array was `[user(prompt)]`.

**Impact.** Every multi-turn conversation on the `conversational-sales-agent` since the Era-46 Nova deploy (`09:58:11Z`) ‚Äî not company-specific. The bot could open a conversation but never continue one. Only reason it wasn't caught earlier: Incident #1 (IAM) was masking ALL turns until ~10:56Z, and the first post-IAM-fix multi-turn conversation (11:06‚Äì11:08) is what exposed it.

**Fix (`src/services/providers/BedrockNovaProvider.js`).** After `_coalesce()`, a new `_stripLeadingAssistant()` step removes any leading `assistant` turn(s) so the array always begins with `user` (Converse's hard requirement). Only the *leading* assistant is dropped ‚Äî interior alternation and the trailing user prompt are untouched, so at most the turn-1 opener's echo is lost, never current context. Empty-after-strip (all-assistant input ‚Äî unreachable via AIService, which always appends a user prompt) degrades to a single prompt-only user turn with a logged warning rather than sending an empty array (which Converse also rejects) or throwing. Unit tests added: the exact production shape `[assistant,user,assistant,user,user]`, the turn-1 empty-history regression guard, and the all-assistant fallback.

**Verification.** Real Bedrock, before/after: the raw assistant-first shape returns `ValidationException: A conversation must start with a user message`; the same shape through the fixed provider returns a real reply; a real 3-turn conversation (turn 2 with an assistant-first history) replies on all three turns. Full unit suite green (the one unrelated `documentExtraction` PDF-fixture failure is pre-existing/environmental). Full production webhook‚ÜíWhatsApp round-trip to be confirmed post-deploy.

**NEW CHECKLIST ITEM (provider migrations):**
> When switching LLM providers, **diff the message-array validation rules between providers explicitly** ‚Äî do not assume the old provider's tolerances hold on the new one. Bedrock Converse enforces at least two rules the Anthropic Messages API did not: (a) the array must **start with a user message** (assistant-first history is rejected), and (b) **no consecutive same-role turns** (must strictly alternate). Enumerate these differences and add a normalization + unit test for each BEFORE the switch, using a realistic multi-turn history shape, not just a single-turn call.

## INCIDENT #3 ‚Äî Prompt Management "Publish" silently saved an empty addendum: two disconnected text sources (2026-07-15)

**Severity:** Production, admin-facing, customer-affecting-by-design (the addendum is meant to shape the AI's live conversational behavior). Silent data loss ‚Äî no error shown, the admin believed the exact text they tested went live. Fixed same day, held for review.

**Symptom.** Viir typed a 267-character conversational-sales-agent prompt addendum, clicked **"Run Test"** (which showed the correct text and passed compliance 4/5 with the one known guarantee false-positive), then clicked **"Publish"**. Version history then showed *"Version 2 ¬∑ Live ¬∑ (empty)"* ‚Äî the saved version had **no text** even though the addendum box on screen still showed the typed content.

**Root cause ‚Äî two independently-sourced text fields, bridged only by an optional intermediate action.** "Run Test" and "Publish" read the addendum text from **different sources**:
- **Run Test** sent the on-screen text in its own request body ‚Äî `testMutation`'s `mutationFn: () => testPromptAddendum(text)` (`PromptManagementTab.tsx:42`) ‚Üí `/prompt-addendum/test` reads `req.body.text` (`aiAdmin.js:222`). It tested the correct 267-char text.
- **Publish** sent **no text at all** ‚Äî `publishMutation`'s `mutationFn: publishPromptAddendum` (bare, `PromptManagementTab.tsx:52-53`), invoked as `publishMutation.mutate()` with no argument (`:127`); `publishPromptAddendum()` sent a hardcoded empty body `JSON.stringify({})` (`api.ts:138-139`); and `/prompt-addendum/publish` **ignored the body entirely** and re-read `candidateText = cfg.draftText ?? ''` from DynamoDB (`aiAdmin.js:249`).

The only writer of a non-empty `draftText` was the **"Save Draft"** button (`/prompt-addendum/draft`), which the user never clicked. Critically, "Run Test" does **not** persist its tested text into `draftText` either ‚Äî its update expression uses `draftText = if_not_exists(draftText, :empty)` (`aiAdmin.js:233`), preserving the existing (empty) value. So at publish time `draftText` was `""`; publish re-tested the empty string (an empty addendum is trivially compliant ‚Üí `allPassed: true`), and stored `text: ""` into the version row (`:267`) and `activeText: ""` into CURRENT (`:277`).

**The client-side `canPublish` gate created false confidence.** `canPublish = liveResult?.allPassed === true && testedText === text` (`PromptManagementTab.tsx:29`) correctly validated the *on-screen* text and enabled the button ‚Äî but the server published a *different* value (`draftText`), so a green gate never implied the right text would go live. The two only coincided if the admin separately clicked "Save Draft" first, which was neither required nor enforced.

**Why the compliance gate didn't catch it.** The gate is a safety filter (does the reply trip a guardrail?), not an equality check. An empty addendum produces perfectly compliant replies, so it passed ‚Äî the gate had no reason to object to publishing nothing.

**Impact.** The defect applied to **every publish for every company** since the feature's first use (2026-07-08) ‚Äî publish never carried the on-screen text, so any admin who didn't happen to click "Save Draft" before publishing stored an empty addendum. Confirmed against live `business_metrics` via full scan of all `CONFIG#PROMPTADDENDUM#*` records: **`viir_trading` is the only tenant that has ever used the feature**, and **both** of its published versions are empty ‚Äî `VERSION#000001` (`text: ""`, published 2026-07-08) and `VERSION#000002` (`text: ""`, published 2026-07-15) ‚Äî with `CURRENT.activeText: ""` throughout. Live conversation turns read `activeText` (`ConversationalAgentService.js:225`) and only inject the addendum when non-empty (`aiConfig.js:421-424`), so **the conversational-sales-agent has never actually had any custom addendum applied** ‚Äî the feature has been a silent no-op since it shipped. No other company has any `PROMPTADDENDUM` records, so none is affected.

**Fix (Option 1 ‚Äî single source of truth, server re-verified).**
- Frontend: `publishMutation`'s `mutationFn` is now the closure `() => publishPromptAddendum(text)` (mirrors `testMutation` exactly), and `publishPromptAddendum(text)` sends `{ text }` in its body ‚Äî the same shape as `testPromptAddendum` (`PromptManagementTab.tsx:52-56`, `api.ts:138-142`). Publish now carries the exact text the gate validated.
- Backend: `/prompt-addendum/publish` no longer reads `cfg.draftText`. It validates `req.body` with `promptAddendumDraftSchema` (400 on missing/extra keys ‚Äî a strict schema, so no forged fields), **re-runs the compliance test server-side on that received text** (the sole authorization to go live ‚Äî a client-reported pass has no channel and is never trusted), and on a genuine pass persists that text as the version row, `activeText`, **and** `draftText` (one value `:t` drives both, so active/draft can't drift) (`aiAdmin.js:244-298`). The old `draftText = :empty` clear is gone.
- The `canPublish` gate is retained as a frontend UX nicety only; real enforcement is the server-side re-test on the received text.

**Data cleanup ‚Äî NOT auto-fixed, human decision required.** Both `viir_trading` versions are genuinely empty. The two empty version-history rows are historical fact and can stay. The open decision is only whether to re-publish the intended addendum going forward ‚Äî that's Viir's call and simply means typing it and publishing again on the fixed flow. No other tenant needs any action.

**Validation.** `tests/aiAdmin.test.js` updated + extended (all green): publish persists the exact `req.body` text as `activeText` (not `draftText`-sourced, not empty), reproduces the prod state (empty stored draft + correct body text) and proves the body text wins, asserts `activeText`/`draftText` are driven by one `:t` value with no `:empty`, blocks (422) on a genuine server-side test failure even for a well-formed publish the client believed passed, 400s a bodyless publish (the fallback-to-draft path is now structurally impossible), and 400s a body carrying extra keys. "Save Draft" behavior is unchanged and its existing tests still pass. Full backend suite: 1662/1663 (the one failure is the pre-existing/environmental `documentExtraction` PDF-under-Jest fixture, documented above ‚Äî unrelated). Dashboard `tsc --noEmit` clean.

**NEW CHECKLIST ITEM (applies to any preview/test-then-commit UI):**
> Any feature with a preview/test step and a separate publish/save step must operate on the **exact same data by construction** ‚Äî never two independently-sourced fields bridged only by an optional intermediate action. The thing you tested and the thing you commit must be the *same value threaded through both paths*, not two reads of storage that a user action is assumed to have synchronized. And the commit step must **re-verify server-side on the exact payload it is about to persist** ‚Äî a client-reported "it passed" is never authorization to write.

**Reference:** `src/routes/aiAdmin.js` (`/prompt-addendum/publish`); `dashboard/src/components/v3/ai-admin/PromptManagementTab.tsx`; `dashboard/src/lib/ai-admin/api.ts`; `src/services/ConversationalAgentService.js:225`; `src/config/aiConfig.js:421-424`; `19_DECISION_LOG.md` Phase 2A / PR 2.

## INCIDENT #4 ‚Äî whatsapp_conversation_started ‚Üí start_ai_conversation silently failed for every unknown contact: no leadPK in the trigger's execution context (2026-07-15)

**Severity:** Production-path, customer-facing feature (not yet live to real customers ‚Äî caught during Viir's live workflow testing). Silent: the customer tapped a welcome button, WhatsApp confirmed the tap, and nothing came back ‚Äî no AI reply, no error. Fixed same day, held for review.

**Symptom.** In the new `whatsapp_conversation_started` workflow "testing now" (a Send-Buttons welcome node ‚Üí per-button branches ‚Üí `start_ai_conversation`), the customer tapped **"Open Demat"**. The button-reply webhook arrived and the paused workflow resumed correctly to the `start_ai_conversation` node ‚Äî which then threw and the AI never engaged. Reproduced twice on viir_trading test contacts: `01:32:00.925Z` and again `01:39:02.062Z`, both `‚öÝÔ∏è AutomationEngine: node "start_ai_conversation" failed in "testing now": start_ai_conversation: leadPK required`.

**Two wrong hypotheses, explicitly ruled out by the investigation (do not re-chase these):**
- *Button-reply parsing bug* ‚Äî NO. `parseButtonReply` reads `msg.interactive.button_reply.id` correctly (`whatsapp.js:1134-1137`) and the webhook passes it to `resumeOnButtonReply`; the resume demonstrably worked (it reached the node). `resumeOnButtonReply` receives an already-parsed id and never touches the interactive payload.
- *Trigger-type mismatch (`new_contact` stored vs `whatsapp_conversation_started` searched)* ‚Äî NO. There is **no `new_contact` trigger type anywhere** in the codebase (grep-confirmed); the save path, the fire path (`whatsapp.js:1854`), the search (`hasActiveWorkflow`, `automations.js:37,51`), and the actual stored DynamoDB record all use the single value `whatsapp_conversation_started`. "New WA Conversation" is only that trigger's UI display label.

**Root cause ‚Äî `start_ai_conversation` needs a `leadPK` its own trigger never provides.** The `whatsapp_conversation_started` trigger fires only from the webhook's unknown/first-contact (`INBOX#`) branch, with a context of `{ phone, name, source, tags }` and **no `leadPK`** (`whatsapp.js:1854-1855`). That leadPK-less context is frozen into the `AUTO_WAIT` record when the workflow pauses at the send-buttons node and restored verbatim on resume, so `start_ai_conversation` hit its precondition `if (!leadPK) throw` (`AutomationEngine.js:803`). No lead was ever created in between: the autonomous `maybeStart` path ‚Äî which *does* resolve-or-create a CRM lead via CIS before engaging ‚Äî is deliberately suppressed whenever an active `whatsapp_conversation_started` workflow exists (`whatsapp.js:1773-1774`, "the workflow OWNS first-contact"), and the workflow's own node assumed a lead that was never made.

**Impact ‚Äî 100% failure for this flow.** Every `whatsapp_conversation_started` ‚Üí `start_ai_conversation` execution for an unknown contact (i.e. every genuinely-new WhatsApp contact ‚Äî the exact audience this trigger exists to greet) failed at the node. Confirmed against live CloudWatch (`/aws/lambda/vt-employee-bot-api`), reproduced twice. Known-lead contexts (e.g. the `keyword_message` path) were unaffected because they carry a real `leadPK`.

**Architectural decision (recorded).** The fix could live in three places: inside the node (as first proposed), at the trigger-fire point (mirroring the `inbound_webhook` route, `automations.js:584-602`), or inside `startForLead`. AutomationEngine.js's header declares *"ADR-013: never creates customers; reads existing leads only"* (`:25`). To preserve that reader boundary, customer-creation was placed in **`ConversationalAgentService.startForLead`** (which already owns AI-conversation lead creation via `maybeStart`), NOT in AutomationEngine. This keeps AutomationEngine a pure reader; it simply passes `leadPK` through (undefined for a conversation_started context).

**Fix.**
- `ConversationalAgentService.startForLead` now treats `leadPK` as **optional**: after the `cfg.enabled` gate (so a disabled company never creates a lead), when `leadPK` is absent it does `CustomerIdentityService.resolveOrCreate({ phone: phone10, name, source: 'whatsapp', skipAutoAssign: true }, { createdBy: 'webhook' })` ‚Äî the exact CIS path `maybeStart` uses ‚Äî deriving the leadPK and handling CIS's contract that `result.lead` is set only on a fresh create (an enriched/idempotent hit is read back by `leadId`). `skipAutoAssign: true` is essential: without it a newly-created lead could be auto-assigned to a human and then the existing `assignedTo` no-hijack guard would immediately bail.
- `AutomationEngine`'s `start_ai_conversation` node drops `if (!leadPK) throw`; it now requires only `phone` (the one field a lead can be resolved from) and passes `leadPK` (possibly undefined) straight through. AutomationEngine's "reads existing leads only" boundary is unchanged.

**Explicitly accepted limitation (product-owner decision, 2026-07-15):** `startForLead` does **not** replicate `maybeStart`'s `updateLeadLastMessage(created)` stamp on a fresh create. The 2026-07-15 adversarial review flagged and verified this as a real medium regression: `CIS._createCustomer` never sets `lastMessageAt`, so if the first AI turn fails to *send* (an `AIService.generate` error or a `WhatsAppSendService` failure ‚Äî note a guardrail block still sends `HANDOFF_MESSAGE`, so it is NOT a trigger), the freshly-created lead is briefly invisible in every `lastMessageAt`-gated read (inbox list, `/my-work` urgentReplies, LeadScoringService recency, auto-assign eligibility) ‚Äî and, because the inbox phone-dedup suppresses the `INBOX#` record for a phone that also has a `LEAD#`, invisible as an unknown contact too ‚Äî until the customer sends a second message (which lands directly in the `LEAD#` partition). For companies with an active `whatsapp_conversation_started` workflow this is the *sole* creation path (maybeStart is suppressed), so the 2026-07-08 protection is absent for them.

Decision: **accept it, do not stamp.** The only text available to stamp here is the synthetic turn-0 seed (the admin's `contextHint`, e.g. "Demat", or the "Hi" fallback) ‚Äî not a genuine customer message. Writing that as `lastMessagePreview`/`lastInboundAt` on the `LEAD#` would misrepresent a synthetic seed as something the customer actually sent, which the product owner judged a worse, more permanent data-integrity cost than the narrow, transient invisibility window (which only opens when an AI first-turn *send* fails immediately after a first-contact create, and which self-heals the moment any real message arrives). Revisit only if a genuine first-inbound text becomes available to `startForLead` at call time.

**Validation.** `tests/conversationalAgentService.test.js` ‚Äî 5 new `startForLead`-without-leadPK tests: CIS create-then-engage (asserts `skipAutoAssign`/`source`/`createdBy`), an enriched CIS hit read back by `leadId`, no-lead-created-when-AI-disabled (cfg gate before CIS), no-hijack of an already-assigned lead, and neither-leadPK-nor-phone returns false. `tests/automationEngine.test.js` ‚Äî the old "throws when leadPK missing" test replaced with "hands off with no leadPK (passes `undefined` through, does not throw)" plus a "throws only when phone is also missing" guard test. Two further tests added after the 2026-07-15 adversarial review closed its two verified test-gap findings: a created-branch `leadPK`-derivation pin (`conversationalAgentService.test.js`) and an engine-level proof that a tapped `send_list` row id resumes a paused wait (`automationEngine.test.js`). Full backend suite: 1674/1675 (the one failure is the pre-existing/environmental `documentExtraction` PDF-under-Jest fixture, documented above ‚Äî unrelated).

**Reference:** `src/services/ConversationalAgentService.js` (`startForLead`); `src/services/AutomationEngine.js` (`start_ai_conversation` case, `:25` boundary); `src/routes/whatsapp.js:1773-1774,1854-1855`; `src/routes/automations.js:584-602` (the CIS-in-route precedent); `src/services/ConversationalAgentService.js` `maybeStart` (the mirrored path); `19_DECISION_LOG.md`.

## INCIDENT #6 ‚Äî Extracted-but-not-recalled: the AI persisted qualification signals to the lead but never read them back, so it re-asked what the customer already answered (2026-07-15)

**Severity:** Quality / customer-experience, customer-facing (not yet live to real customers ‚Äî caught in Viir's live test). **Root cause ‚Äî a write-only extraction loop.** `_applyExtractedSignals` persists the AI's own extracted `productInterest` / `expectedValue` / `closureDeadline` onto the `LEAD#` record ([ConversationalAgentService.js:235-263](src/services/ConversationalAgentService.js#L235-L263)) ‚Äî but nothing ever read those fields *back* into the prompt. `_runTurn`'s generate context carried only `latestMessage` + free-text `conversationHistory` + settings/knowledge ‚Äî **no structured qualification state** ([:432-447](src/services/ConversationalAgentService.js#L432-L447)). Compounding it, the button-tap seed was never even written to `MSG#` history: `resolveForLead` updates only the `CONV#` *lastMessage preview*, not a `MSG#` record ([conversationResolver.js:277,303,327](src/utils/conversationResolver.js#L277)), so `_fetchConversationHistory` never contained "Demat" as a message ‚Äî the only lingering trace was the AI's own turn-1 reply phrasing. Net effect: the model had to **re-infer** `productInterest` from free text every turn, and inconsistently dropped it by turn 4. This is a *class* ‚Äî every extracted signal (button-tap OR organic mid-conversation) was equally un-recalled; the organic case was just masked because the customer's real message stays in history.

**Design investigation (two options weighed).** Option A = write the seed as a real inbound `MSG#` (fixes only the conversation-start case, needs transcript-honesty tagging + per-source wiring, has a 20-message-window caveat). Option B = inject a structured "known qualification state" block into the prompt (source-agnostic, closes the extraction‚Üírecall loop for button-tap AND organic AND any future source, 2 backend files, no frontend/schema change). **Chose B.**

**Requirement-5 design decision ‚Äî PROVISIONAL framing (the load-bearing part).** A structured "KNOWN SO FAR" block carries a real stale-state-override risk: if presented as immutable fact, the model could stay anchored to "demat" when the customer later says *"actually, mutual funds instead."* This is compounded by `_applyExtractedSignals`'s additive-union merge ([:240](src/services/ConversationalAgentService.js#L240)), which never drops a superseded interest ‚Äî so the stored `productInterest` genuinely accumulates `["demat", "mutual funds"]`. Handled by **framing, not assertion**: the block is rendered as *"things mentioned EARLIER ‚Äî treat as PROVISIONAL, not confirmed fact; the CUSTOMER'S MOST RECENT MESSAGE below is always authoritative; if it changes/narrows/contradicts anything here, follow the newer message,"* and it is placed **above** the latest-message section so precedence reads correctly. This matches the agent's existing compliance-conservative "ask directly, don't guess" stance and makes the pre-existing *"never re-ask something they already told you"* instruction ([aiConfig.js:445](src/config/aiConfig.js#L445)) *enforceable* (supplies the structured facts) rather than duplicating it.

**Fix.** `_buildKnownState(lead)` reads the **same** lead fields `_applyExtractedSignals` writes ‚Äî no new storage ‚Äî and `_runTurn` passes it as `context.knownState` ([ConversationalAgentService.js](src/services/ConversationalAgentService.js)); `aiConfig.js`'s conversational-sales-agent template renders the PROVISIONAL `KNOWN SO FAR` block, above the authoritative latest-message section. The `company-phone-index` GSI is `ProjectionType: ALL`, so the `lead` `continueTurn` already receives carries these attributes (no extra read). `null`/empty renders nothing ‚Äî byte-identical to before for turn 0 and any conversation with no captured signals.

**Explicitly accepted limitation (disclosed).** This closes the *read-back* gap only. The *write-side* additive merge is out of scope ("no new storage mechanism, just close the read-back gap") ‚Äî so the stored/blocked `productInterest` can still *list* a superseded interest after a correction; the PROVISIONAL framing prevents the AI **anchoring** to it, but the data itself isn't de-duplicated. If the stored state should also drop a replaced interest, that's a separate write-side change (teach extraction to emit replacements, or bound/most-recent the merge) ‚Äî optional follow-up.

**Validation.** `tests/aiConfig.test.js` ‚Äî the block renders `productInterest`/amount/timeline, is framed PROVISIONAL and defers to the latest message, sits above `CUSTOMER'S MOST RECENT MESSAGE`, renders partial state correctly, and is byte-identical to baseline when absent/empty. `tests/conversationalAgentService.test.js` ‚Äî (a) button-tap round-trip (seed ‚Üí extracted+persisted ‚Üí recalled next turn), (b) organic mid-conversation extraction recalled a later turn, (c) correction/req-5 (stale `knownState` passed *alongside* the corrective latest message, not instead of it), and the empty-baseline case. Full backend suite: 1683/1684 (the one failure is the pre-existing/environmental `documentExtraction` PDF-under-Jest fixture, documented above ‚Äî unrelated).

**Reference:** `src/services/ConversationalAgentService.js` (`_buildKnownState`, `_runTurn` context, `_applyExtractedSignals`); `src/config/aiConfig.js` (conversational-sales-agent `promptTemplate`, `KNOWN SO FAR` section); `src/utils/conversationResolver.js:277` (why the seed never reached `MSG#`); INCIDENT #4 (the `start_ai_conversation` flow this surfaced from).

## INCIDENT #7 ‚Äî AI occasionally re-asks name/city it already has: a RARE stochastic instruction-following miss (NOT deterministically verifiable) (2026-07-15)

**Read this first ‚Äî how this differs from #6.** INCIDENT #6 was a *deterministic data gap* (a field was genuinely never read back; always reproducible; unit-testable). This one is **the opposite**: the data was present the whole time (name/city sit in the conversation transcript the model already receives), and the AI *usually* uses them correctly ‚Äî it just occasionally re-asks anyway. It is a **rare stochastic instruction-following miss**, and per the verification note below it **cannot be given a clean "fixed and confirmed" before/after number at reasonable cost**. Do not read this entry as "fixed and confirmed" ‚Äî it is "mitigated on sound reasoning, regression-checked for no harm."

**Symptom.** After the customer gave their name and city early in a conversation, the agent sometimes re-asked them several turns later (e.g. *"What's your name and city, Ravi?"* ‚Äî asked while already using the name "Ravi"). Reported from live testing.

**Investigation ‚Äî reproduced, but rare.** Live Bedrock-Nova runs (real `AIService.generate`, `conversational-sales-agent`, viir_trading's actual settings) across three failure-shape scenarios, 50 runs total: the re-ask reproduced cleanly **once** ‚Äî *"Awesome, let's get started. What's your name and city, Ravi? And how urgently are you looking to invest‚Ä¶"* ‚Äî but only **1/50 (~2%)**. The other 49 correctly acknowledged name/city and moved to the next unanswered step. So it is real but low-frequency, not systematic.

**Root cause ‚Äî an asymmetry in the qualification script, not a data gap.** viir_trading's `qualificationRules` reads: *"1) Skip re-asking if they already stated their interest ‚Äî acknowledge and move on. 2) Collect name and city. 3) Ask urgency‚Ä¶ 4) Ask capital‚Ä¶"* Step 1 has an explicit skip-if-present guard; **step 2 ("Collect name and city") does not** ‚Äî it instructs collection unconditionally. Combined with the base template's never-re-ask line, which listed *"(goal, budget, timeline)"* and **omitted name/city**, the two fields most likely to be re-asked were exactly the two nothing told the model to skip.

**Verification note (verbatim, why there is no clean before/after number).** *You asked for a live test that "confirms whether the strengthened instruction actually prevents the re-ask." At a ~2% base rate, a live A/B can't do that at any practical scale: to detect a reduction from ~2% to ~0% with confidence you'd need roughly 150‚Äì300 runs per arm, and even then a probabilistic model can't be proven to never slip. A fixed-prompt run of 20‚Äì40 will read 0 re-asks ‚Äî but so does the baseline most of the time, so it proves nothing.* This is the honest reason there is no "fixed and confirmed" claim here: the base rate is too low for an A/B at feasible cost, and a stochastic model behavior can't be proven eliminated.

**What was done (two levels, same gap).**
- **Template (`src/config/aiConfig.js`, conversational-sales-agent `promptTemplate`) ‚Äî applies to every company:** (Item 1) the never-re-ask line now names `name`/`city` explicitly and calls them out as the worst offenders; (Item 2) a new universal skip-if-present imperative near GOAL ‚Äî *"treat every question as 'ask ONLY if it isn't already answered above'‚Ä¶ skip to the next UNanswered step‚Ä¶ never ask for a name or city that already appears above"* ‚Äî which overrides any step-by-step script (admin or default) that lists a collect-step without a guard.
- **Config (`CONFIG#CONVPROMPT#viir_trading.qualificationRules`) ‚Äî this company now, while the template rolls out:** added the missing skip clause to step 2, mirroring step 1's phrasing.

**Regression check (what a live test CAN show, and did).** With the strengthened template, live runs confirmed no harm in both directions: (A) name/city already given ‚Üí **14/14** progressed to the next unanswered step, 0 re-asks, 0 freezes; (B) name/city genuinely absent ‚Üí **8/8** still asked for them (the fix does not over-suppress ‚Äî it doesn't make the agent afraid to ask legitimately). This proves the change doesn't break the ~98% that already worked or paralyze the agent; it does **not**, and cannot, prove the ~2% miss is eliminated.

**Reference:** `src/config/aiConfig.js` (never-re-ask line + the skip-if-present GOAL imperative); `CONFIG#CONVPROMPT#viir_trading` (`qualificationRules` step 2); INCIDENT #6 (the extracted-but-not-recalled fix ‚Äî contrast: deterministic and testable, unlike this one).

**Priority:** Mitigated (template + config), regression-checked for no harm. Explicitly NOT "confirmed fixed" ‚Äî a rare stochastic miss with no feasible clean before/after measurement. If it's still observed in production at a bothersome rate, the next lever is not more prompt wording but a structural one (e.g. surfacing collected identity as an explicit prompt fact, or a large-N A/B if a measured number is genuinely needed).

## continueTurn ‚Äî Concurrent Inbound Messages Race the aiTurnCount Read/Increment (Burns Turn Budget, Duplicate Replies)

**Issue (found 2026-07-15 while tracing the viir_trading re-ask conversation, NOT fixed):** `ConversationalAgentService.continueTurn()` does a non-atomic read-modify-write on the conversation's `aiTurnCount`: it reads `turnCount = conversation.aiTurnCount` (`ConversationalAgentService.js` ~L722), runs the whole `_runTurn()` (generate + send + audit), then calls `ConversationService.incrementAiTurn()` (`_runTurn` ~L507) at the very end. When a customer sends two messages a couple of seconds apart, Meta fires two separate webhook invocations that both look up the lead and both call `continueTurn` before either has incremented ‚Äî so both read the SAME `aiTurnCount` and both run as the same logical turn.

**Live evidence (lead `c7e62725`, conv `conv_01KX81MXSDB1TES0XS5ZJ5EVB8`, phone 919353266686):** the customer sent `"Sip"` at 11:29:28 and `"1"` at 11:29:30 (2s apart). Both processed as **turn 4** ‚Äî two `ai_conversation_turn` audit records (11:29:32.466 and 11:29:34.777, both `guardrailTripped=false`), producing two near-duplicate outbound replies 2.3s apart: *"Got it, you'd like to explore a SIP in mutual funds... could you please share your name and city?"* and *"Got it, you're planning a SIP. What's your name and city?..."*. Net effect: one logical step consumed **two** of the conversation's turns.

**Why it matters:** `MAX_TURNS = 5` is deliberately tight (halved 10‚Üí5 in the 2026-07-14 cost trial). Each spurious duplicate turn (a) sends the customer a redundant message, (b) permanently consumes 1 of only 5 turns, and (c) pulls the `turn_limit_reached` handoff forward ‚Äî in this conversation it contributed directly to the agent running out of turns and handing off before ever capturing name/city. It also doubles the LLM spend for that step. Webhook retries and out-of-order delivery compound the same window.

**Fix (not scoped/implemented):** make the turn advance atomic and gate on it rather than trusting a pre-read count. Options: (a) an atomic DynamoDB `ADD aiTurnCount :1` with `ConditionExpression` on the expected prior value, using the returned new value to detect and drop the loser of the race before spending an `AIService.generate()` call; (b) per-conversation serialization/debounce of inbound text so back-to-back messages coalesce into one turn; (c) short idempotency/lock window keyed on `conversationId`. Any of these must still let genuinely distinct sequential turns proceed. Note the near-identical read-then-increment shape also exists conceptually for the double-send it produces ‚Äî a fix here removes both the duplicate reply and the double turn-budget spend.

**Priority:** Medium ‚Äî not a crash and no data corruption, but it actively wastes a scarce (5-turn) budget and sends duplicate customer-facing messages on a completely normal user behavior (typing two quick messages), making premature turn-limit handoffs materially more likely. Higher real-world impact than most Low items here because the triggering behavior is common, not an edge case.

## CORS origin rejection throws instead of declining cleanly ‚Äî pages Telegram on every disallowed-origin request in production, not just logs (found 2026-07-16, local-dev CORS debugging)

**Issue (found while debugging a local-dev CORS failure, NOT fixed).** `src/app.js`'s CORS setup (`:54-60`) rejects a disallowed origin by calling `callback(new Error(\`Origin ${origin} not allowed by CORS\`))`:
```js
const corsMiddleware = cors({
  origin: (origin, callback) => {
    if (!origin || allowedOrigins.includes(origin)) return callback(null, true);
    callback(new Error(`Origin ${origin} not allowed by CORS`));
  },
  credentials: true,
});
```
That is a real `Error`, not the `cors` package's own documented rejection convention (`callback(null, false)`).

**Root cause ‚Äî confirmed against the actual installed `cors@2.8.6` source** (`node_modules/cors/lib/index.js`), not assumed from docs:
```js
originCallback(req.headers.origin, function (err2, origin) {
  if (err2 || !origin) {
    next(err2);
  } else {
    ...
  }
});
```
A truthy `err2` (our thrown `Error`) makes the package call `next(err2)` ‚Äî routing straight into Express's error-handling chain instead of the package's own clean-decline path (set no `Access-Control-Allow-Origin` header, end the response). This happens identically for the OPTIONS preflight and any actual request from a disallowed origin. It lands in `src/middleware/errorHandler.js:27`: `res.status(err.status || 500).json({ error: err.message || 'Internal server error', ... })` ‚Äî the thrown `Error` has no `.status` set, so every disallowed-origin request gets a **500**, not the semantically-correct 403/clean-decline.

**Why this matters beyond local-dev friction.** `errorHandler.js:24` calls `logger.error('Error occurred', err)` on this path before the default-500 response ‚Äî and per this codebase's own established logging convention (`logger.error`/`logger.alert` page Telegram, `logger.warn` doesn't ‚Äî see the embed-timeout-fix entry elsewhere in this doc), **every rejected-origin request currently pages Telegram in production**, indistinguishable from a genuine server error. Any stray/scanning/misconfigured client hitting the production API with an unrecognized `Origin` header generates a production page, not a log line ‚Äî live paging-noise risk, not just a local-dev inconvenience (this is how the bug was found: a local dev server on an unlisted port hit exactly this path).

**Fix direction (not implemented in this pass).** Change the rejection to `callback(null, false)` instead of `callback(new Error(...))`. With `origin: false`, the `cors` package resolves the preflight/request cleanly (no `Access-Control-Allow-Origin` header, browser reports a CORS failure client-side) without ever calling `next(err)` ‚Äî no `errorHandler` involvement, no 500, no Telegram page. `app.js` was deliberately not touched in this pass ‚Äî this is local-dev CORS debugging that surfaced a separate, real production issue, not something to fix as a drive-by.

**Reference:** `src/app.js:54-60` (the `corsMiddleware` definition); `src/middleware/errorHandler.js:24,27` (the paging + default-500 behavior); `node_modules/cors` v2.8.6 `lib/index.js` (confirmed rejection-propagation mechanism).

**Priority:** Worth a real look, not just local-dev friction ‚Äî this pages on every rejected-origin production request today. Low implementation cost (one-line change, `new Error(...)` ‚Üí `false`), but flagging for deliberate review rather than a drive-by fix since it touches production error-handling/alerting behavior directly.

Option (b) is probably the smaller, more consistent fix given the consume side only ever intends to correlate against known leads.

**Reference:** `src/services/AutomationEngine.js`'s `send_flow` case (`target` construction); `src/services/WhatsAppSendService.js`'s `resolveContact()` (`{ phone }` ‚Üí `INBOX#` fallback); `src/routes/whatsapp.js`'s `sendRegisteredFlow()` (marker write) and the `nfm_reply` consume site inside the webhook handler's `if (lead) { ... if (isNewMsg) { ... } }` block (consume, lead-scoped only).

**Priority:** Low ‚Äî narrow edge case (a Flow sent via the AutomationEngine `send_flow` node to a phone number with no CRM lead yet is uncommon; the Inbox manual-send route and welcome-button follow-ups both always resolve a real `leadPK` first). Worth closing before the `flow_completed` trigger consumes `flowId` broadly, so a known gap doesn't surface as a confusing "why is this always null for X contacts" support question later.

## `flow_completed` trigger inherits the unknown-contact correlation gap above (found 2026-07-17, wiring the flow_completed trigger)

**Issue.** The `flow_completed` trigger (`src/routes/whatsapp.js`'s `nfm_reply` handler, fired via `AutomationEngine.fireTrigger(webhookCompanyId, 'flow_completed', ctx)`) only fires inside the known-lead branch (`if (lead) { ... if (isNewMsg) { ... if (matchedFlowId) { ... } } }`), and `matchedFlowId` can only ever resolve from the same `PENDINGFLOW#` correlation logic described in the entry directly above. That entry already establishes the correlation never runs at all on the unknown-contact (`INBOX#`) path ‚Äî this is the same root cause, not a new one: a Flow response from a contact with no CRM lead yet will always carry `flowId: null` and therefore can never fire `flow_completed`, regardless of how a workflow's trigger config (`{flowId}` filter or company-wide catch-all) is set.

**Fix:** identical to the fix above ‚Äî this entry closes automatically whenever that one is fixed. Not tracked as a separate action item.

**Reference:** see `send_flow`'s `{ phone }` fallback entry above; same file/branch references apply, plus the `flow_completed` fire site inside the same `if (matchedFlowId)` block this gap already governs.

**Priority:** Low ‚Äî same root cause and same narrow-edge-case reasoning as the entry above; this is a note that the boundary also applies to the newer trigger, not an independent gap.

## Flow builder option IDs can be manually duplicated within one component, unvalidated (found 2026-07-17, building the option-id auto-derive fix)

**Issue (found while building the deriveOptionId auto-derive fix for RadioButtonsGroup/CheckboxGroup/Dropdown options, not fixed ‚Äî flagged as a known scope boundary, not silently expanded).** `OptionsListEditor`'s per-option "Edit ID" affordance (`dashboard/src/components/flow-builder/componentEditors.tsx`) lets an admin hand-set an option's `id` directly, sanitized to a safe character set (lowercase letters/digits/underscore) but never checked against the OTHER options already in the same `dataSource` array. Nothing stops two options within one Dropdown/RadioButtonsGroup/CheckboxGroup from ending up with the identical `id` ‚Äî auto-derive itself can never produce this (`deriveOptionId()`'s own `_2`/`_3` suffixing guarantees uniqueness against `otherIds` on every keystroke), but a manual edit bypasses that guarantee entirely, silently.

**Effect:** Meta's `data-source` array would contain two `{id, title}` entries with the same `id`. Downstream behavior on Meta's side of a duplicate-id data-source is not something this fix scoped out ‚Äî at minimum it's a wasted/confusing option; at worst a customer's selection becomes ambiguous in the stored response (which of the two identically-`id`'d options did they actually pick?). No UI signal exists today to warn the admin this happened ‚Äî unlike field names (cross-screen duplicate check, `NameField`'s inline error state) or screen ids (`validateFlow()`'s duplicate-screen-id check), there is no equivalent check for option ids.

**Fix (not implemented):** add a duplicate-option-id check, either inline in `OptionRow`'s "Edit ID" field (matching `NameField`'s own `duplicate` prop + inline red error text, using the same `otherIds` list `OptionRow` already receives ‚Äî no new data plumbing needed) or as a new case in `validateFlow()` (`dashboard/src/types/flowBuilder.ts`), matching how that function already flags cross-screen duplicate field names. The inline `NameField`-style check is likely the smaller, more consistent fix given `otherIds` is already threaded through to exactly this component.

**Reference:** `dashboard/src/components/flow-builder/componentEditors.tsx`'s `OptionRow` (the unvalidated "Edit ID" input) and `deriveOptionId()`/`otherIds` (the collision set already available but only enforced on the auto-derive path); `dashboard/src/types/flowBuilder.ts`'s `validateFlow()` (existing precedent for a cross-component/cross-screen duplicate-id check, not yet extended to options).

**Priority:** Low ‚Äî requires a manual, deliberate "Edit ID" edit into a colliding value (auto-derive itself can never produce this), so it's a narrow admin-error case rather than something that can happen by just using the builder normally. Worth closing before this "Edit ID" affordance sees real usage at scale, so a confusing double-attributed response doesn't surface as a support question later.
